  # 2. 项目目标

  本系统用于帮助已注册用户管理本人研讨室预约流程，在用户授权和官方接口允许范围内实现：

  1. 本站邮箱注册、登录、找回密码和账号注销。
  2. 用户手动提交官方系统的 `reflushToken`。
  3. 后端自动将 `reflushToken` 置换为新的 `accessToken` 和新的 `reflushToken`。
  4. 后端通过新的 `accessToken` 获取用户学号与真实姓名，完成本站账号和官方身份绑定。
  5. 定期刷新并安全持有用户官方会话凭证。
  6. 查看未来三天内可预约的研讨室及剩余时间段。
  7. 手动预约符合规则的研讨室。
  8. 创建未来自动预约任务，并在目标日期进入官方可预约窗口后自动提交预约。
  9. 支持多人研讨室联约、邀请确认和预授权自动联约。
  10. 预留用户到场后自动签到逻辑。
  11. 自动执行签退任务。
  12. 提供管理员后台、审计日志、任务状态查看和异常处理能力。

  # 3. 官方授权边界

  需求方确认，本系统已获得官方系统或相关应用的正式开发及第三方接入授权，授权范围包括：

  - 保存用户的 `reflushToken` 与 `accessToken`；
  - 后台自动刷新用户凭证；
  - 后台自动发起预约；
  - 在满足官方到场条件后执行签到；
  - 自动签退；
  - 邀请其他学生参与预约；
  - 使用用户广场展示已注册用户必要身份信息。

  ## 3.1 签到边界

  自动签到不是远程代签到。

  签到必须满足：

  1. 主预约人实际到达研讨室现场。
  2. `systemMac` 与 `qrSignCheckCode` 由官方授权流程生成或取得。
  3. 系统只在获得本次有效签到参数后调用官方签到接口。
  4. 在签到参数取得逻辑尚未开发前，系统仅保留签到任务、接口适配器和状态流，不实际执行签到请求。

  ## 3.2 签退边界

  需求方确认，官方对签退位置无限制，只需要在允许条件下点击签退按钮。因此，本系统可以在预约结束前自动提交签退请求。

  ## 3.3 多人联约边界

  副预约人仅在以下任一条件满足时，才可以被加入预约请求中的 `members` 数组：

  1. 副预约人针对本次邀请主动点击“接受”；或
  2. 副预约人提前在本站开启“允许被自动联约”，且提交预约时该授权仍有效。

  不得将“未回复邀请”自动视为同意。

  # 5. 核心术语

  | 术语                   | 定义                                                 |
  | ---------------------- | ---------------------------------------------------- |
  | 本站用户               | 在本系统使用邮箱注册的用户                           |

  | 主预约人               | 创建并提交预约请求的本站用户                         |
  | 副预约人               | 被加入多人预约 `members` 数组中的参与者              |
  | `accessToken`          | 官方业务接口调用所需的短效访问凭证                   |
  | `reflushToken`         | 官方接口中使用该拼写的滚动刷新凭证                   |

  | 官方业务 API Client ID | 图书馆业务接口使用的 UUID 形态 `appId`               |

  # 6. 已确认业务规则

  ## 6.1 可预约日期范围

  用户可以查看并预约：

  - 今天；
  - 明天；
  - 后天。

  即官方系统存在包含当天在内的三日预约窗口。

  用户允许提前创建更远日期的自动预约任务，但系统只能在目标日期进入官方可预约窗口后真正提交预约。

  ## 6.2 每日预约次数与时长

  每个用户每天遵守以下规则：

  | 规则                 | 限制     |
  | -------------------- | -------- |
  | 每日最大预约次数     | 2 次     |
  | 单次最大预约时长     | 120 分钟 |
  | 每日累计最大预约时长 | 240 分钟 |

  系统必须进行双层校验：

  1. 前端填写预约或自动任务时进行即时提示和阻止；
  2. 后端提交官方预约接口前重新进行强校验。

  官方接口返回结果始终作为最终判定结果。

  ## 6.3 预约时间粒度

  自动预约任务只能使用以下开始时间和结束时间：

  - 整点，例如 `17:00`；
  - 半点，例如 `17:30`。

  即时间值必须满足：

  ```text
  分钟部分只能为 00 或 30
  ```

  待确认：

  - 手动预约是否同样必须限制为整点或半点；
  - 最短预约时长是否应统一限制为房间字段 `reservationMinTime` 返回值。

  ## 6.4 房间人数规则

  | 房间类型           | 最低总人数 | 规则                          |
  | ------------------ | ---------- | ----------------------------- |
  | 4 人及以下房间     | 1 人       | 允许主预约人单独预约          |
  | 5 人及以上可用房间 | 3 人       | 必须包含主预约人在内至少 3 人 |
  | 8 人研讨间         | 不可预约   | 前端隐藏或置灰，后端禁止提交  |
  | 12 人研讨间        | 不可预约   | 前端隐藏或置灰，后端禁止提交  |

  已确认：

  ```text
  minReservationNum = 3 包含主预约人
  ```

  因此，对返回 `minReservationNum = 3` 的可用大房间：

  ```text
  members 数组至少需要包含 2 名副预约人
  ```

  系统实现时优先使用官方房间字段 `minReservationNum` 进行校验，同时叠加产品规则中对 8 人、12 人房间的禁用限制。

  # 7. 应用标识与服务端配置

  ## 7.1 微信 OAuth 应用标识

  ```env
  WECHAT_APP_ID=wxc8cbe72d481f23db9
  ```

  已确认：

  - 该值只属于学校在微信开放平台登记的 OAuth 应用；
  - 其作用是发起微信扫码授权或微信内静默授权；
  - 它与图书馆业务接口没有关系；
  - V1 不接入微信 OAuth，因此 V1 不需要使用该配置完成任何业务请求。

  ## 7.2 图书馆业务 API Client 配置

  ```env
  LIBYY_APP_ID=41043f17-3c17-4f2e-894c-5d615f992db9
  LIBYY_APP_SECRET=<SERVER_SIDE_SECRET>
  ```

  已确认：

  - UUID 形态的 `LIBYY_APP_ID` 是官方图书馆业务接口调用时必须使用的应用标识；
  - 后端执行凭证刷新、身份获取和业务接口调用时均使用该应用标识；
  - `LIBYY_APP_SECRET` 只能由 Cloudflare Worker 服务端读取；
  - 前端不得获得或拼接 `LIBYY_APP_SECRET`；
  - 文档、前端代码、公开仓库和日志中不得出现真实 `LIBYY_APP_SECRET`。

  ## 7.3 服务端秘密配置

  生产环境至少需要以下配置：

  ```env
  LIBYY_APP_ID=41043f17-3c17-4f2e-894c-5d615f992db9
  LIBYY_APP_SECRET=<SECRET>
  
  TOKEN_ENCRYPTION_KEY=<SECRET>
  SESSION_SECRET=<SECRET>
  PASSWORD_HASH_SECRET=<OPTIONAL_SECRET>
  
  SMTP_HOST=<TO_BE_CONFIRMED>
  SMTP_PORT=<TO_BE_CONFIRMED>
  SMTP_SECURE=<TO_BE_CONFIRMED>
  SMTP_USERNAME=<SECRET_OR_CONFIG>
  SMTP_PASSWORD=<SECRET>
  SMTP_FROM_NAME=<TO_BE_CONFIRMED>
  SMTP_FROM_ADDRESS=<TO_BE_CONFIRMED>
  
  APP_BASE_URL=<PRODUCTION_DOMAIN>
  ADMIN_INITIAL_EMAIL=<TO_BE_CONFIRMED>
  ```

  要求：

  - 生产敏感项必须通过 Worker 服务端秘密配置注入；
  - 本地开发配置文件不得提交版本库；
  - 前端环境变量中不得出现任何官方密钥或用户 token；
  - 所有日志必须对敏感数据脱敏。

  # 8. 总体技术架构

  ## 8.1 架构组成

  ```text
  用户浏览器
     │
     ▼
  Cloudflare Pages 前端
     │ HTTPS / 本站 Session
     ▼
  Cloudflare Workers API
     ├── D1：用户、任务、预约、邀请、状态、审计
     ├── Secret：官方密钥、SMTP 密码、加密密钥
     ├── 定时任务：刷新、预约扫描、签退执行
     └── 官方图书馆 API
  ```

  ## 8.2 前端职责

  前端负责：

  - 注册、登录、找回密码和注销交互；
  - 引导用户提交 `reflushToken`；
  - 展示绑定状态和凭证异常提示；
  - 房间及可用时间展示；
  - 手动预约表单；
  - 自动预约任务配置；
  - 候选房间顺序调整；
  - 用户广场；
  - 邀请接受或拒绝；
  - 自动联约设置；
  - 到场签到入口占位；
  - 用户历史预约和任务结果展示；
  - 管理员后台页面。

  前端不得：

  - 保存官方 `accessToken` 或 `reflushToken` 到 LocalStorage；
  - 直接调用需要 `LIBYY_APP_SECRET` 的官方 API；
  - 展示完整官方 token；
  - 绕过后端权限检查提交预约或管理请求。

  ## 8.3 后端职责

  Workers 后端负责：

  - 本站认证和会话校验；
  - 邮箱验证码及密码找回；
  - 官方 token 置换、加密存储和滚动刷新；
  - 官方用户身份绑定；
  - 官方房间信息转发和标准化；
  - 手动预约执行；
  - 自动预约调度；
  - 多人邀请状态判定；
  - 邮件发送；
  - 签到逻辑保留与后续接入；
  - 签退执行；
  - 管理员后台 API；
  - 幂等控制、并发控制、重试和审计日志。

  ## 8.4 数据库职责

  D1 负责保存：

  - 本站账户；
  - 官方身份绑定关系；
  - 加密后的官方凭证；
  - 房间查询缓存或同步记录；
  - 手动及自动预约任务；
  - 候选房间列表；
  - 邀请关系；
  - 自动联约授权设置；
  - 预约执行记录；
  - 签到签退任务状态；
  - 邮件验证码及密码重置记录；
  - 系统审计日志。

  # 10. 本站账户体系

  ## 10.1 注册方式

  用户使用邮箱注册本站账号。

  已确认规则：

  - 注册必须通过邮箱验证码；
  - 只允许中国国内主流邮箱服务商；
  - 一个邮箱只能创建一个本站账户；
  - 一个学号只能绑定一个本站账户；
  - 不允许用户修改绑定邮箱；
  - 允许用户注销账户；
  - 支持找回密码。

  ## 10.2 邮箱白名单

  邮箱域名应通过服务端配置维护，例如：

  ```json
  [
    "qq.com",
    "163.com",
    "126.com",
    "yeah.net",
    "sina.com",
    "outlook.com"
  ]
  ```

  以上列表仅为配置结构示例，最终允许域名清单待需求方确认。

  ## 10.3 注册流程

  ```text
  用户输入邮箱
  → 后端检查邮箱域名是否允许
  → 后端发送邮箱验证码
  → 用户提交验证码和密码
  → 后端校验验证码
  → 创建未绑定官方身份的本站账户
  → 用户登录
  → 进入官方凭证绑定引导页
  ```

  ## 10.4 密码要求

  待确认后固定以下配置：

  - 最低密码长度；
  - 是否强制包含数字、字母和特殊字符；
  - 登录错误次数限制；
  - 验证码有效期；
  - 找回密码链接有效期；
  - 是否支持验证码登录。

  ## 10.5 注销流程

  注销功能必须包含：

  ```text
  用户发起注销
  → 再次验证登录身份或邮箱验证码
  → 禁用本站会话
  → 停止所有未来自动任务
  → 停止凭证刷新
  → 删除或不可逆销毁官方 token
  → 根据数据保留策略处理历史预约、邀请和审计记录
  ```

  历史业务记录及审计日志是否保留、保留多久，待确认。

  # 11. 官方凭证绑定流程

  ## 11.1 用户输入内容

  V1 用户只需要提交：

  ```text
  reflushToken
  ```

  用户不需要提交：

  - `accessToken`；
  - 学号；
  - 姓名；
  - `appId`；
  - `appSecret`。

  ## 11.2 正确绑定链路

  已确认的首次绑定链路为：

  ```text
  用户提交有效 reflushToken
  → Worker 调用官方 /reflushToken 接口
  → 官方返回新的 accessToken、新的 reflushToken 和 expires
  → Worker 使用新的 accessToken 调用官方 /user 接口
  → 官方返回 userId 与 realName
  → Worker 校验本站邮箱与学号唯一绑定约束
  → Worker 加密保存新的 accessToken 和新的 reflushToken
  → 完成官方身份绑定
  ```

  不得实现为：

  ```text
  用户提交 reflushToken
  → 直接将其作为 accessToken 请求 /user
  ```

  因为该流程不符合已确认链路。

  ## 11.3 绑定请求内部流程

  本站接口建议定义为：

  ```http
  POST /api/v1/credentials/bind
  Content-Type: application/json
  Cookie: <SITE_SESSION>
  ```

  请求体：

  ```json
  {
    "reflushToken": "<USER_SUBMITTED_REFLUSH_TOKEN>"
  }
  ```

  后端处理步骤：

  1. 校验用户已登录本站。
  2. 校验用户当前没有处于封禁或注销状态。
  3. 校验请求中存在非空 `reflushToken`。
  4. 不记录明文 token 日志。
  5. 使用用户提交的 `reflushToken` 调用官方刷新接口。
  6. 若刷新失败，向用户返回绑定失败原因，不写入凭证数据。
  7. 若刷新成功，解析新的 `accessToken`、新的 `reflushToken` 和 `expires`。
  8. 使用新的 `accessToken` 调用官方用户身份接口。
  9. 解析 `userId` 与 `realName`。
  10. 校验该学号是否已与其他本站账户绑定。
  11. 若校验通过，将身份与加密凭证写入数据库。
  12. 将用户凭证状态设为 `ACTIVE`。
  13. 记录脱敏审计事件。
  14. 返回绑定成功结果。

  ## 11.4 绑定时学号冲突的特殊风险

  由于系统必须先刷新 `reflushToken` 才能取得用于查询身份的新 `accessToken`，在发现“该学号已被其他邮箱绑定”之前，用户提交的旧 `reflushToken` 已经可能被官方滚动替换。

  因此，以下情况必须在正式开发前确定处理规则：

  ```text
  用户 A 登录本站
  → 提交属于已绑定学号 B 的有效 reflushToken
  → 系统刷新成功得到新的 token
  → 系统查询身份后发现学号 B 已被另一邮箱账户占用
  ```

  不得简单丢弃刷新后得到的新 token，否则可能导致原绑定账户失去可继续刷新的凭证。

  该冲突处理策略列为上线前阻塞项。

  # 12. 官方凭证自动刷新

  ## 12.1 刷新周期

  已确认：

  ```text
  每隔 90 分钟，系统自动刷新所有活跃且已绑定官方凭证的用户。
  ```

  当前官方成功响应显示：

  ```text
  expires = 7200 秒
  ```

  即当前样本中的访问凭证有效期为 120 分钟。90 分钟刷新策略为业务要求，可为访问凭证预留缓冲时间。

  ## 12.2 滚动刷新特征

  已确认：

  ```text
  旧 reflushToken 在滚动刷新成功后立即作废。
  ```

  因此，每次刷新必须同时处理：

  - 新 `accessToken`；
  - 新 `reflushToken`；
  - 新有效期；
  - 旧凭证失效风险；
  - 并发重复刷新风险；
  - 写库失败导致凭证丢失风险。

  ## 12.3 刷新执行规则

  每个用户的刷新任务必须满足：

  1. 同一用户在任一时刻只允许存在一个刷新操作。
  2. 自动刷新、用户主动业务请求前刷新、管理员触发刷新不得并发执行。
  3. 调用官方刷新接口前必须取得该用户凭证刷新锁。
  4. 刷新成功后必须将新 `accessToken` 与新 `reflushToken` 一并写入。
  5. 不允许只更新其中一个 token。
  6. 写入成功后才可释放刷新锁。
  7. 日志不得包含完整 token。
  8. 刷新失败必须记录官方状态码与业务错误码。
  9. 若明确返回登录失效，应将用户凭证状态更新为 `REAUTH_REQUIRED`。
  10. 若为网络错误或系统内部异常，是否重试须按后续失败策略确认。

  ## 12.4 凭证状态

  ```text
  UNBOUND          用户尚未绑定官方身份
  ACTIVE           凭证正常，可执行业务请求
  REFRESHING       当前正在执行凭证刷新
  REFRESH_FAILED   最近一次刷新失败，但尚未确认需要重新绑定
  REAUTH_REQUIRED  官方明确提示登录失效，需要用户重新粘贴凭证
  DISABLED         用户注销、被封禁或官方绑定被管理员停用
  ```

  ## 12.5 刷新审计信息

  每次刷新记录：

  - 用户内部 ID；
  - 刷新开始时间；
  - 刷新结束时间；
  - 刷新触发来源：定时任务 / 用户业务前置刷新 / 管理员操作；
  - 是否成功；
  - 官方 HTTP 状态码；
  - 官方业务错误码；
  - 脱敏错误信息；
  - token 版本号变化；
  - 不保存完整 token。

  # 15. 手动预约功能

  ## 15.1 创建流程

  ```text
  用户选择日期
  → 选择房间
  → 选择开始时间与结束时间
  → 系统校验时长和时间合法性
  → 系统根据 minReservationNum 判断是否需要副预约人
  → 如需副预约人，校验邀请接受状态或自动联约授权
  → 后端刷新或验证当前 accessToken 状态
  → 后端调用官方预约接口
  → 保存预约执行结果
  → 向用户展示成功或失败信息
  ```

  ## 15.2 提交前校验

  后端必须校验：

  - 用户已绑定官方身份；
  - 用户未被封禁；
  - 用户凭证处于可用状态；
  - 日期处于官方允许预约窗口；
  - 房间未被产品规则禁用；
  - 单次预约时长不超过 120 分钟；
  - 当日已预约次数不超过 2 次；
  - 当日累计预约时长不超过 240 分钟；
  - 多人房间副预约人数量足够；
  - 每名副预约人已经接受邀请或开启自动联约；
  - 时间段当前仍可预约。

  # 16. 自动预约任务

  ## 16.1 V1 支持策略

  自动预约任务采用：

  ```text
  固定目标日期 + 固定开始时间 + 固定结束时间 + 按顺序尝试的备选房间列表
  ```

  不支持：

  - 自动随机选择任意空闲房间；
  - 自动改变用户指定的日期；
  - 自动改变用户指定的时间；
  - 自动补充未经授权的副预约人。

  ## 16.2 时间限制

  自动任务的开始和结束时间必须为：

  ```text
  HH:00 或 HH:30
  ```

  例如：

  - `08:00` 至 `09:00`；
  - `09:30` 至 `11:00`；
  - `17:00` 至 `19:00`。

  不允许：

  - `08:15`；
  - `10:20`；
  - 其他非半小时边界时间。

  | `target_date`            | 目标日期                 |
  | `start_time`             | 预约开始时间             |
  | `end_time`               | 预约结束时间             |

  | `candidate_rooms`        | 按尝试顺序保存的备选房间 |
  | `member_selection`       | 副预约人列表             |
  | `task_status`            | 当前任务状态             |
  | `attempt_count`          | 已执行尝试次数           |
  | `last_attempt_at`        | 最近尝试时间             |
  | `success_reservation_id` | 成功后的预约订单 ID      |

  | `created_at`             | 创建时间                 |
  | `updated_at`             | 更新时间                 |

  ## 16.4 备选房间执行顺序

  用户可以按照优先顺序配置候选房间：

  ```json
  [
    {
      "priority": 1,
      "roomId": 6,
      "roomName": "7E01"
    },
    {
      "priority": 2,
      "roomId": 7,
      "roomName": "7E02"
    }
  ]
  ```

  执行时：

  1. 系统按 `priority` 从小到大读取候选房间；
  2. 查询目标日期下房间是否已进入官方可预约窗口；
  3. 判断候选房间是否仍可预约；
  4. 判断房间人数规则是否满足；
  5. 尝试提交第一个满足条件的房间；
  6. 若官方明确返回房间冲突，可继续尝试下一个备选房间；
  7. 若由于用户当日次数、时长、凭证失效或成员不足导致失败，不再盲目尝试其他房间；
  8. 成功后立即停止后续尝试。

  ## 16.5 自动任务状态

  ```text
  DRAFT             已创建但未启用
  WAITING_WINDOW    等待目标日期进入可预约窗口
  WAITING_MEMBERS   等待副预约人确认
  READY             条件满足，等待执行
  SUBMITTING        正在提交官方预约请求
  SUCCESS           预约成功
  FAILED            官方请求失败且任务不可继续
  CANCELLED         用户主动取消
  EXPIRED           超过可执行时机仍未成功
  ```

  ## 16.6 开放窗口识别

  目标日期是否可以正式提交预约，不得仅凭系统自行猜测开放时间。

  在官方开放时刻尚未确认前，V1 应以官方房间查询接口是否能够返回目标日期可预约信息作为窗口判定依据：

  ```text
  目标日期已能够通过官方房间接口查询并存在可预约时间片
  → 认为该日期已进入可预约窗口
  ```

  仍需补充确认：

  - 新一天预约窗口每天确切开放时刻；
  - 自动任务扫描频率；
  - 首次提交是否需要尽可能贴近开放瞬间；
  - 房间冲突后是否重试及重试次数。

  # 17. 用户广场与多人联约

  ## 17.1 用户广场目标

  用户广场用于帮助主预约人发现本站中可共同参与预约的其他用户。

  广场只展示已完成官方身份绑定且账号正常的本站用户。

  | ---------------- | ------------------------ |
  | 真实姓名         | 待确认是否完整展示       |
  | 学号             | 待确认是否完整或脱敏展示 |
  | 所属学院         | 待确认是否展示           |
  | 是否允许自动联约 | 待确认是否展示           |
  | 邀请按钮         | 展示                     |

  虽然需求方确认具备展示姓名或学号的授权，但产品层面仍应确定用户是否具有退出广场展示或隐藏部分信息的能力。

  ## 17.3 邀请来源

  主预约人可以通过以下方式选择副预约人：

  1. 从本站用户广场选择；
  2. 使用官方用户搜索接口按学号或姓名查询。

  ## 17.4 自动联约授权

  每个本站用户具有设置：

  ```text
  allow_auto_join_reservation = true | false
  ```

  规则：

  - 用户可以主动开启；
  - 用户可以随时关闭；
  - 仅对创建预约时仍处于开启状态的新预约生效；
  - 关闭后不得再自动加入新的多人预约；
  - 对已经成功创建的预约不自动撤销参与关系。

  默认值待确认，建议默认值为 `false`，由用户主动开启。

  ## 17.5 手动邀请流程

  对于未开启自动联约的用户：

  ```text
  主预约人选择副预约人
  → 创建邀请记录
  → 系统向副预约人发送邀请邮件
  → 副预约人点击接受或拒绝
  → 接受后邀请状态变为 ACCEPTED
  → 拒绝后邀请状态变为 REJECTED
  → 只有 ACCEPTED 才允许加入 members
  ```

  ## 17.6 邀请邮件按钮

  邀请邮件包含：

  - 邀请人姓名；
  - 房间名称；
  - 预约日期；
  - 开始时间；
  - 结束时间；
  - 接受按钮；
  - 拒绝按钮；
  - 邀请有效期说明。

  邮件按钮应携带一次性签名令牌，不得仅凭可猜测 ID 直接接受或拒绝邀请。

  ## 17.7 邀请状态

  ```text
  PENDING        等待响应
  AUTO_APPROVED  用户已提前授权自动联约
  ACCEPTED       用户手动接受
  REJECTED       用户手动拒绝
  EXPIRED        已超时且未响应
  CANCELLED      主预约人取消任务或撤回邀请
  USED           已被成功预约消耗
  ```

  ## 17.8 邀请超时

  原需求中的“4 分钟后自动接受”仅适用于已预授权自动联约的用户，但该类用户实际上无需等待四分钟。

  对于未开启自动联约的用户：

  - 未响应不得自动接受；
  - 四分钟后的处理行为待确认；
  - 可选行为包括：邀请失效、继续等待、提示主预约人更换成员。

  # 22. 任务调度与并发控制

  ## 22.1 定时任务类型

  系统至少需要以下定时任务：

  | 定时任务     | 用途                                   |
  | ------------ | -------------------------------------- |
  | 凭证刷新扫描 | 每 90 分钟刷新活跃用户凭证             |
  | 自动预约扫描 | 检查目标日期是否进入预约窗口并执行预约 |
  | 签到扫描     | 参数接入后，在签到时间执行任务         |
  | 签退扫描     | 在预约结束前执行签退                   |
  | 邀请过期扫描 | 处理已超时邀请                         |
  | 通知补偿扫描 | 处理邮件失败或通知重试                 |

  ## 22.2 凭证刷新并发控制

  由于旧 `reflushToken` 在成功刷新后立即作废，刷新任务必须满足强一致性要求。

  每个用户必须具备：

  - 单用户刷新锁；
  - token 版本号；
  - 刷新开始时间；
  - 刷新锁超时时间；
  - 刷新结果状态。

  禁止发生：

  ```text
  定时刷新读取旧 token
  同时用户预约前刷新也读取同一个旧 token
  两者同时向官方刷新
  其中一个成功后令旧 token 失效
  另一个失败或覆盖数据库中的最新 token
  ```

  ## 22.3 预约提交幂等性

  自动预约任务必须防止重复提交。

  至少使用：

  - 自动任务唯一 ID；
  - `SUBMITTING` 状态锁；
  - 执行次数记录；
  - 成功后立即停止；
  - 同一任务不得被两个 Worker 实例同时提交。

  ## 22.4 签退幂等性

  签退任务必须防止：

  - 用户手动签退与自动签退并发；
  - 定时扫描重复提交；
  - 网络重试造成重复请求。

  具体实现依赖官方签退接口响应结构补证。

  # 26. 安全要求

  ## 26.1 Token 安全

  必须满足：

  - 用户 token 仅通过 HTTPS 提交；
  - token 在数据库中加密保存；
  - token 不保存于浏览器 LocalStorage；
  - token 不返回前端；
  - token 不输出日志；
  - token 不出现在管理员普通页面；
  - 用户注销时销毁 token；
  - 凭证失效后停止自动业务任务。

  ## 26.2 密码安全

  必须满足：

  - 密码仅保存安全哈希；
  - 不保存明文密码；
  - 找回密码使用一次性验证码或一次性链接；
  - 验证码必须具有过期时间和使用次数限制；
  - 登录失败应具备防暴力尝试策略。

  ## 26.3 邀请安全

  必须满足：

  - 邮件接受或拒绝链接使用一次性令牌；
  - 令牌只对应一条邀请；
  - 邀请已处理后令牌失效；
  - 未授权用户不得被加入 `members`；
  - 自动联约设置变更必须记录审计日志。

  ## 26.4 管理员安全

  必须满足：

  - 管理员接口进行独立权限校验；
  - 高风险操作记录审计日志；
  - 默认不展示敏感凭证；
  - 管理员登录安全策略待确认。

  # 27. 异常处理原则

  ## 27.1 凭证失效

  当官方返回：

  ```json
  {
    "msg": "登录已失效",
    "code": 2003
  }
  ```

  系统应：

  1. 标记用户凭证为 `REAUTH_REQUIRED`；
  2. 停止新的自动预约提交；
  3. 停止依赖官方凭证的自动签退尝试；
  4. 向用户发送凭证重新绑定通知；
  5. 在前端显示重新粘贴 `reflushToken` 引导。

  ## 27.2 token 格式错误

  当官方返回：

  ```json
  {
    "msg": "token不正确 无法解析",
    "code": 2004
  }
  ```

  绑定阶段应：

  1. 拒绝绑定；
  2. 不保存用户提交的 token；
  3. 提示用户重新复制有效 `reflushToken`。

  ## 27.3 自动预约失败

  自动预约失败需区分：

  | 失败类型             | 处理方式                 |
  | -------------------- | ------------------------ |
  | 第一个房间被占用     | 尝试下一个备选房间       |
  | 所有候选房间均不可用 | 任务失败并通知用户       |
  | 人数不足             | 不继续提交，提示补充成员 |
  | 凭证失效             | 停止任务，要求重新绑定   |
  | 当日次数超限         | 任务失败，不继续尝试     |
  | 当日时长超限         | 任务失败，不继续尝试     |
  | 网络异常             | 重试规则待确认           |

  ## 27.4 自动签退失败

  自动签退失败处理规则：

  - 失败日志；
  - 任务写入 `official_response_redacted`；
  - 保持可重试状态，等待下一轮 Cron；
  - 避免重复签退。

  # 28. 测试与验收要求

  ## 28.1 账户模块验收

  - 允许域名邮箱可以注册；
  - 非允许域名邮箱被拒绝；
  - 邮箱验证码有效；
  - 重复邮箱不可注册；
  - 登录、退出和找回密码正常；
  - 用户无法修改绑定邮箱；
  - 用户可以注销账户。

  ## 28.2 凭证绑定验收

  - 用户提交有效 `reflushToken` 后，系统先执行刷新接口；
  - 系统获得新 `accessToken` 后调用 `/user`；
  - 成功解析 `userId` 和 `realName`；
  - 新 token 被加密保存；
  - 无效 token 返回明确提示；
  - 已失效 token 返回重新绑定提示；
  - token 不出现在前端和日志中。

  ## 28.3 凭证刷新验收

  - 定时刷新每 90 分钟执行；
  - 刷新成功后两类 token 均更新；
  - 同一用户并发刷新被阻止；
  - 旧 token 作废情况下不会覆盖新 token；
  - `code=2003` 可正确标记重新认证；
  - 管理员可查看脱敏刷新状态。

  ## 28.4 房间浏览验收

  - 用户可查看今天、明天、后天房间；
  - 页面显示容量、最低人数和剩余时段；
  - 8 人、12 人房间不可提交预约；
  - 可按时段和人数筛选；
  - 房间接口异常时页面有明确提示。

  ## 28.5 手动预约验收

  - 单人房间可单人预约；
  - `minReservationNum = 3` 时必须有两名有效副预约人；
  - 不满足人数条件时禁止提交；
  - 单次时长超过 120 分钟被阻止；
  - 每日次数和累计时长超限被阻止；
  - 成功预约能够保存订单记录。

  ## 28.6 自动预约验收

  - 可创建固定日期和固定时段任务；
  - 开始结束时间只能为整点或半点；
  - 可设置多个按序备选房间；
  - 房间冲突时按顺序尝试下一个候选房间；
  - 成功后停止后续尝试；
  - 失败后保存失败原因；
  - 副预约人授权不足时不得提交。

  ## 28.7 邀请验收

  - 可从本站广场选择用户；
  - 可通过官方搜索添加用户；
  - 未开启自动联约者必须接受邀请；
  - 已开启自动联约者可直接满足联约条件；
  - 未响应邀请不会自动接受；
  - 接受与拒绝邮件链接一次性有效。

  ## 28.8 签到验收

  - 签到任务可以创建；
  - 签到状态可以展示；

  - 已签到和已取消状态幂等处理。

  ## 28.9 签退验收

  - 到达结束前 10 分钟后自动执行；
  - 已签退不会重复执行；
  - 失败可记录并进入下一轮重试；
  - 预约订单与签退记录正确关联。

  - 是否需要权限分级；
  - 是否允许人工触发业务任务；
  - 是否需要管理员二次验证。

  ## 29.9 重复学号绑定冲突策略

  必须确定：

  ```text
  当用户提交一个有效 reflushToken，系统已刷新生成新 token 后，
  才发现该官方学号已经绑定至另一个本站账户时，
  新生成 token 如何安全处理。
  ```

  该问题直接关系到凭证安全和原账户可用性，属于生产上线阻塞项。

  # 31. 当前最终确认结论

  截至 v1.0，以下规则视为已确认：

  1. V1 使用邮箱注册登录，不接入微信 OAuth 或 CAS 登录。
  2. 用户粘贴的是 `reflushToken`。
  3. 正确绑定链路为：`reflushToken → /reflushToken → 新 accessToken → /user → userId 与 realName`。
  4. 官方业务 API 使用 UUID 形态 `LIBYY_APP_ID`。
  5. 微信 `WECHAT_APP_ID` 与图书馆业务接口完全独立。
  6. 官方请求头使用 `Authorization: {TOKEN}`，不加 `Bearer`。
  7. 刷新成功响应字段位于根节点：`accessToken`、`reflushToken`、`expires`。
  8. 用户信息成功响应字段位于根节点：`userId`、`realName`。
  9. 旧 `reflushToken` 在滚动刷新成功后立即作废。
  10. 凭证必须加密存储，且同一用户刷新必须防并发。
  11. 用户可查看今天、明天、后天的研讨室。
  12. 每日最多预约 2 次，单次最长 120 分钟，每日合计最长 240 分钟。
  13. `minReservationNum = 3` 包含主预约人，因此需要两名副预约人。
  14. 8 人、12 人研讨间不可预约。
  15. 自动预约任务采用固定日期、固定时间段和按顺序尝试的备选房间。
  16. 自动预约时间必须位于整点或半点。
  17. 未开启自动联约的副预约人必须手动接受邀请。
  18. 已开启自动联约的用户可以在授权有效时直接加入预约。
  19. 签到仅允许在用户到场且官方生成参数后执行，当前默认只承诺 room 2 映射。
  20. 签退目标规则为结束前 10 分钟自动执行，并通过官方历史接口回读确认。
